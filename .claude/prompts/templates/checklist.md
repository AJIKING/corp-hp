# プロンプト作成チェックリスト

実装依頼プロンプトを書いたら、投げる前にこのリストで確認する。

---

## 構造

- [ ] **冒頭3行以内**で「何をするか」が伝わるか
- [ ] 変更不可の前提条件が**1箇所にまとまっている**か（複数セクションに散らばっていないか）
- [ ] 実装タスクが**依存順**に並んでいるか（スキーマ → ロジック → UI → テスト の順など）
- [ ] 仕様詳細（レイアウト・レスポンシブ仕様など）は**末尾の参照セクション**に分離されているか

## 情報の過不足

- [ ] **似ている既存実装のパス**が書いてあるか（既存パターンに合わせるために必須）
- [ ] UI 仕様がある場合、**レスポンシブ対応**について触れているか
- [ ] CLAUDE.md に書いてある内容と**重複する注意事項を書いていない**か（「TDD で進めろ」「シングルクォートを使え」等の詳細は不要）

## テスト

- [ ] **テスト方針**が層ごとに書かれているか
  - Unit: 何をテストするか
  - Component: 何をテストするか
  - E2E: 何をテストするか
- [ ] 不要な層にも**「不要」と理由を明記**しているか（書かないと勝手に作る）
- [ ] 「CLAUDE.md の TDD ルールに従う」と**参照を入れている**か

## 読みやすさ

- [ ] 1文に情報を**詰め込みすぎていない**か（1文1情報が理想）
- [ ] 「背景・経緯・意思決定の理由」と「作業指示」が**混在していない**か
- [ ] タスクが大きすぎないか（PR として読みやすい単位に分割されているか）
